SentinelNet v2 is a production-oriented AI-powered Network Intrusion Detection System built on the CICIDS2017 dataset. The project has been implemented phase-by-phase from raw multi-file ingestion through model training, anomaly detection, ensemble inference, explainability, real-time replay, alerting, dashboarding, API deployment, performance optimization, and advanced response automation.

The repository now contains executable pipelines for all 16 requested phases, persistent artifacts for every major stage, operational logs, unit and integration tests, a Streamlit dashboard, a FastAPI service layer, a production React SOC frontend, performance-tuned serving defaults, and a Phase 16 advanced response layer for zero-day candidate triage, simulated auto-blocking, feature-drift monitoring, and continuous learning.

2. Project Objective

The project objective was to build a modular and deployable AI-powered Network Intrusion Detection System that can:

- Process multiple CICIDS2017 CSV files reliably
- Detect malicious traffic in real time
- Classify known attack types
- Surface zero-day anomaly candidates
- Generate operational alerts with risk scores
- Explain predictions and model behavior
- Replay traffic as a real-time simulation
- Expose serving and monitoring interfaces for deployment
- Provide a professional SOC web dashboard connected to the FastAPI backend

9. Known Warnings and Runtime Notes

The following warnings have appeared during real runs and are currently non-blocking:

- scikit-learn model-version mismatch warnings when loading older serialized estimators with a newer library version
- XGBoost serialized-model compatibility warnings for older artifact formats
- TensorFlow retracing warnings during repeated prediction calls
- Native Windows TensorFlow GPU warning for TensorFlow >= 2.11; CPU execution is expected unless using WSL2 or TensorFlow-DirectML
- Browser/dev-server polling can create many GET /stream log lines when multiple frontend tabs are open

These warnings do not indicate pipeline failure, but they should be cleaned up for long-term maintainability by standardizing the serialization toolchain and retraining or re-exporting older artifacts where practical.
